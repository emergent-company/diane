# Full Chat: Interactive Chat in the Companion App

**Date:** 2026-04-30  
**Author:** DianeCoder  
**Rev:** 2 (corrected architecture — agent pipeline, not just graph writes)

## Goal

Enable the companion app to start new chat sessions and send messages interactively. The message must flow through Diane's agent processing pipeline (same as the Discord bot) — not just be stored in the MP graph.

## Architecture

### Current (broken for chat): message → MP graph store → nobody processes it

```
Companion App                 diane serve (local_api.go)           MP Cloud
    │                               │                                │
    ├─ POST /api/sessions/{id}/messages ──→ AppendMessage() ──────→ MP Graph
    │                                                                   │
    │                                                              ┌─────┘
    │                                                              ▼
    │                                                         Message stored,
    │                                                         NO agent processes it
    │                                                         (no assistant response)
```

### Corrected: message → agent pipeline (same as Discord bot)

```
Discord Bot Flow:
  onMessageCreate() → handleMessage() → triggerAgentWithContext()
    → CreateSession if new
    → Find agent definition on MP ("diane-default")
    → CreateRuntimeAgent(definitionID)
    → buildTriggerPromptWithTodos() — adds session context + todos
    → TriggerAgentWithInput(agentID, prompt, sessionID) — MP cloud runner
    → Poll for completion (2s, 120s timeout)
    → Fetch run messages
    → Delete runtime agent
    → Return response text → send to Discord thread

Required for Companion App Chat:
  POST /api/chat/send → NEW handler mimicking triggerAgentWithContext()
    → Same pipeline as above, but:
      - No Discord reactions (👀, ✅, 🛑)
      - No threads
      - Session is tied to the chat, not a Discord channel
      - Runs synchronously (one request, full response back)
```

## Key Insight: Go-side Change Required

The previous design plan was wrong — it assumed the existing `POST /api/sessions/{id}/messages` was sufficient. It is NOT. We need a **new Go endpoint** that drives the full agent pipeline.

## Implementation Plan

### Phase 1: New Go Endpoint — `POST /api/chat/send`

**File:** `server/cmd/diane/local_api.go` (~150 lines new)

```go
// Request
type chatSendRequest struct {
    SessionID string `json:"session_id,omitempty"` // empty = create new
    Content   string `json:"content"`
    AgentName string `json:"agent_name,omitempty"` // "diane-default" default
}

// Response
type chatSendResponse struct {
    SessionID string            `json:"session_id"`
    RunID     string            `json:"run_id"`
    Messages  []messageJSON     `json:"messages"` // all messages including the response
    Success   bool              `json:"success"`
    Error     string            `json:"error,omitempty"`
}
```

**Handler flow** (modeled on `bot.go` `triggerAgentWithContext` at line 2048):

1. Parse request body
2. Create or reuse session
3. Append user message to session (`bridge.AppendMessage`)
4. Find default agent definition (`bridge.ListAgentDefs` → find by agentName)
5. Create runtime agent (`bridge.CreateRuntimeAgent`)
6. `defer` cleanup: `bridge.Client().Agents.Delete(ctx, agentID)`
7. Build trigger prompt with session context + recent messages
8. Trigger agent (`bridge.TriggerAgentWithInput`)
9. Poll for completion (2s intervals, 120s timeout, cancellable)
10. Fetch run messages (`bridge.GetRunMessages`)
11. Return response with all messages

**Route registration:**
```go
mux.HandleFunc("/api/chat/send", api.handleChatSend)
```

**Timeout consideration:** The handler is synchronous and may take 30-120s. The companion app's HTTP client has a 10s timeout. Two options:

**Option A — Extend companion app timeout** (simpler):
- Change the send call to use a 180s timeout specifically for chat send
- Single request/response, no polling on the Swift side

**Option B — Async + Polling** (more complex, matches Discord bot):
- Returns `{session_id, run_id}` immediately (202 Accepted)
- Companion app polls `GET /api/sessions/{id}/messages` (already exists)
- Show "Thinking..." state while polling

**Recommendation:** Start with **Option A** (synchronous, extended timeout). If latency is an issue, switch to async+polling.

### Phase 3: Companion App — API Client Methods

**File:** `Sources/CompanionCore/DianeAPIClient.swift`

Add `sendChatMessage()`:
```swift
// Send a chat message and wait for the full agent response
func sendChatMessage(sessionID: String?, content: String, agentName: String = "diane-default") async throws -> ChatSendResponse {
    let body: [String: Any] = [
        "session_id": sessionID as Any,
        "content": content,
        "agent_name": agentName
    ]
    let jsonData = try JSONSerialization.data(withJSONObject: body)
    let data = try await post("/api/chat/send", body: jsonData, timeout: 180)
    return try JSONDecoder().decode(ChatSendResponse.self, from: data)
}

// Models
struct ChatSendResponse: Codable, Sendable {
    let sessionID: String
    let runID: String
    let messages: [DianeMessage]
    let success: Bool
    let error: String?
}
```

**Timeout fix:** The existing `post()` method uses a hardcoded 10s timeout. Need to either:
- Add an optional `timeout` parameter to `post()`, or
- Create a separate `postWithTimeout()` method

~10 lines of changes to `DianeAPIClient.swift`.

## Open Questions

1. **Concurrent sends to the same session** — the Discord bot has an `activeChans` guard per channel. What's the equivalent for chat sessions? Options:
   - One-in-flight per session (lock)
   - Queue messages per session
   - Shortcut: Companion app can just disable the send button while waiting

2. **Session title** — the Discord bot names sessions "Discord #channel". What should Chat sessions be named? Options:
   - User-provided (prompt for title on first message)
   - Auto-generated from first message content (same as Discord thread naming)
   - "Chat #N" with incrementing counter

3. **Agent selection** — should the chat always use "diane-default", or allow selecting a different agent? For MVP, hard-code to "diane-default".

4. **Error recovery** — if the agent run crashes/polls out, the session still has the user's message but no assistant response. Should we show an error bubble in the chat? Retry?
