# Diane Memory System

How Diane remembers, recalls, consolidates, and forgets — powered by the emergent.memory knowledge graph.

## Data Model

Diane stores three kinds of things in the knowledge graph:

| What | Graph Type | Properties | Auto-Embedding |
|------|-----------|------------|----------------|
| **Conversation turn** | `Message` | `role`, `content`, `sequence_number`, `session` | ✅ content gets vector embedding |
| **Conversation session** | `Session` | `name`, `started_at`, `ended_at`, `message_count`, `status` | ✅ summary gets embedding |
| **Durable fact** | `MemoryFact` | `content`, `confidence`, `memory_tier`, `relevance_score`, `status` | ✅ content gets embedding |

Relationships:
```
Session ──has_message──→ Message[]
Session ←─extracted_from── MemoryFact[]
MemoryFact ──related_to──→ MemoryFact[]
```

## What Happens When

### User sends a message

```
User types → Diane gets message via MCP tool
  │
  ├─ 1. Store message as Message object in graph
  │     (auto-embedding generated by platform on content)
  │
  └─ 2. Run lightweight fact detector (LLM call, ~50 tokens)
         │
         ├─ Correction detected?     → Update existing MemoryFact (confidence=1.0)
         ├─ New preference detected? → Create MemoryFact (confidence=1.0)
         ├─ Discovery detected?      → Create MemoryFact (confidence=0.7)
         └─ Nothing notable?         → Skip
```

**Cost:** 1 API call to store + 1 small LLM call per turn

### Session ends

```
User disconnects / timeout / explicit "/end"
  │
  └─ Session-end hook fires
      │
      ├─ 1. Fetch all messages for this session
      │     (graph.list type=Message, filter session=<key>)
      │
      ├─ 2. Run full-transcript LLM extraction
      │     ├─ Extract entities and facts
      │     ├─ Generate one-paragraph session summary
      │     └─ Wire relationships: fact → extracted_from → session
      │
      ├─ 3. Update Session object
      │     ├─ ended_at = now
      │     ├─ message_count = total
      │     └─ summary = LLM output
      │
      └─ 4. Bump relevance scores
             For each existing fact referenced in this session:
             relevance_score += 0.1
```

**Cost:** 1 API call to list messages + 1 LLM call (~2-4K tokens) + N API calls to create facts

### Nightly dreaming runs

```
Cron trigger (configurable, e.g. 3 AM)
  │
  └─ Dreaming pipeline (Tier 3)
      │
      ├─ 1. Fetch unconsolidated sessions
      │     (graph.list type=Session, filter status=completed, exclude label "dreamed")
      │
      ├─ 2. For each session:
      │     ├─ Deep extraction (re-run with more context)
      │     ├─ Cross-reference with existing MemoryFacts
      │     └─ Merge duplicates (keep higher confidence)
      │
      ├─ 3. Confidence decay
      │     for each active MemoryFact:
      │       days_old = now - created_at
      │       new_confidence = confidence × 0.5^(days_old / 30)
      │       if relevance_score > 0.7:
      │         new_confidence += 0.1
      │       if new_confidence < 0.2:
      │         status = "archived"
      │       elif new_confidence > 0.9:
      │         status = "reinforced"
      │       else:
      │         update confidence = new_confidence
      │
      ├─ 4. Pattern detection
      │     Fetch all active MemoryFacts
      │     Run LLM cluster analysis
      │     Wire related_to edges between related facts
      │
      ├─ 5. Label processed sessions with "dreamed"
      │
      └─ 6. Generate consolidation report
```

**Cost:** N LLM calls (1 per session + 1 pattern analysis) + M API calls for decay updates

> **Note on decay updates:** Without Issue #180 (filter-then-action bulk pipeline), the decay step requires listing all active MemoryFacts, iterating through them, and updating each one individually. With #180, it becomes a single API call.

### User resumes a session

```
User returns after hours/days
  │
  └─ Session resume
      ├─ 1. Load recent session summary (last session)
      ├─ 2. Fetch high-confidence facts (confidence > 0.7)
      │     (graph.list type=MemoryFact, filter confidence>0.7, status=active)
      ├─ 3. Fetch facts accessed in last 7 days (reinforcement signal)
      └─ 4. Inject into context:
          └─ "Previous session: <summary>. Known facts: <top-5 facts>."
```

### User asks "remember that thing about X"

```
User: "What did I say about database pooling?"
  │
  └─ Semantic memory recall
      ├─ 1. Hybrid search across all Message + MemoryFact objects
      │     (memory.query --mode=search "database pooling")
      │
      ├─ 2. Rank results by:
      │     ├─ Vector similarity (semantic match)
      │     ├─ Lexical match (keyword overlap)
      │     └─ (future) RecencyBoost + AccessBoost — Issue #179
      │
      └─ 3. Return top results with context:
          └─ "In our code review session, you asked about database pooling
               and I said sql.DB handles pooling by default. Configure
               SetMaxOpenConns(25) and SetMaxIdleConns(5)."
```

### User corrects Diane

```
User: "No, I use Neovim not VS Code"
  │
  └─ Correction handling
      ├─ 1. Find existing fact about VS Code
      │     (graph.list type=MemoryFact, filter content="VS Code")
      ├─ 2. Update content + set confidence=1.0
      ├─ 3. Create new fact "User uses Neovim" (confidence=1.0)
      └─ 4. Relate new fact as superseding old one
```

### Stale facts get cleaned up

```
Automatic (during dreaming) or on-demand:
  │
  ├─ Facts with confidence < 0.2 → status = "archived"
  │   (hidden from normal queries, still searchable explicitly)
  │
  └─ Facts archived for > 90 days → pruned (hard deleted)
      (requires client-side iteration without Issue #180)
```

## Platform Dependencies

| Feature | Required For | Status |
|---------|-------------|--------|
| Custom types (Session, Message, MemoryFact) | All storage | ✅ Works without schema registration |
| Auto-embeddings | Semantic search | ✅ Verified (vector similarity returns meaningful results) |
| Hybrid search (FTS + vector) | Memory recall | ✅ `memory.query --mode=search` works |
| **Recency-aware search** | Ranking by recency | ✅ **SHIPPED** in v0.35.214 — `--recency-boost`, `--access-boost`, `--recency-half-life` |
| Property filters (gt/lt) | Decay/pruning queries | ✅ `--filter confidence=0.3 --filter-op lt` works |
| Graph traversal | Session→Message→Fact navigation | ✅ via `GetObjectEdges` with ID-based relationships |
| Vector similarity | Related fact discovery | ✅ `FindSimilar` with distance scores |
| **Bulk filter-then-action (Issue #180)** | Efficient decay/prune pipeline | ❌ Requires client-side iteration today |
| **Native conversation types (Issue #178)** | First-class Session/Message storage | ❌ Workaround: custom types |
| **Project-scoped tokens** | Test/prod data isolation | ✅ Verified |
